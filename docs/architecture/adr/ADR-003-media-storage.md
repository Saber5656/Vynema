# ADR-003: Media Storage - Two R2 Buckets

Status: accepted
Date: 2026-07-03
Issue: #2

## Decision

Use two Cloudflare R2 buckets:

| Bucket | Public? | Purpose |
|---|---|---|
| `vynema-media-pending` | no | Agent upload targets, review previews, rejected evidence, quarantine. |
| `vynema-media-public` | yes after publication | Approved media copied during publication. |

Uploads use presigned S3-compatible PUT URLs generated by the Worker.
Publication performs server-side `CopyObject` from pending to public and then
deletes the pending object when safe.

App servers and Workers never proxy video bytes for upload, playback, or
publication copy.

## Rationale

A private/public bucket split makes pending media non-public structurally true.
The public bucket contains only approved content. R2 avoids egress charges and
fits the no-paid-spend MVP when guarded by application quotas.

## Rejected Alternatives

| Alternative | Reason rejected |
|---|---|
| Single bucket + signed playback URLs | Easier to leak or cache incorrectly; revocation semantics become harder. |
| Worker-proxied upload/playback | Violates the no-byte-proxy rule and stresses Workers CPU/memory limits. |

## Quota Guard

The global active storage cap is 8 GiB, below the 10 GB free-tier storage budget.
Issue #14 enforces storage reservations before upload capability creation.
