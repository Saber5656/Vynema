## 3. Problem（問題点）

### 3.1 DB設計とPerf設計の根本的不一致（最重要 / 繰り返し発生）

**影響**: パイプラインで高優先度3件（PIPE-ISSUE-1/2/3）、検索でも3件（FTS vs pg_trgm、cursor vs offset、SSR vs CSR）が全てDB↔Perf間の不一致。tech-leader と analyzer が独立して設計した結果、同じ機能に対して異なる実装方針を採用していた。

**根本原因**: 並行作業時に中間同期ポイントがなく、レビュー段階で初めて不整合が発覚するパターンが繰り返された。

> 言及: reviewer, analyzer, tech-leader

### 3.2 外部API制約の発見タイミングが遅い

**影響**: Gen-4 Turbo が Image to Video のみという制約が、#13 DB設計完了後に判明し、#23/#24/#25 の3タスクで v2.0 修正が必要になった。

**根本原因**: researcher の調査と tech-leader/designer の設計が並行しており、調査結果が設計に反映される前に設計が完了してしまった。

> 言及: tech-leader, designer, researcher

### 3.3 コスト見積もりの根拠が曖昧

**影響**: 初版で Runway $0.50/5秒という誤った値で設計。リサーチサマリーには正しい値があったがクロスチェック不足。リトライ・失敗時再生成・テスト用生成のオーバーヘッドも未考慮。

> 言及: tech-leader, researcher, analyzer

### 3.4 大規模ファイルの肥大化（未対応）

**影響**: ai-pipeline-data-design.md が 1,500行超、home-design-spec.md が 1,185行、search-design-spec.md が 1,321行。team-config.md §9.1 で分割検討を定めていたが未実施。コンテキスト消費とドキュメント参照効率の両面で非効率。

> 言及: tech-leader, designer

### 3.5 方針決定のブロッキング

**影響**: Gen-4 Turbo の方針判断で project-owner の判断待ちが発生し、#23/#24/#25 が数日間ブロック。エスカレーションパスが明確でなかった。

> 言及: tech-leader, project-manager

### 3.6 共有コンポーネント管理が暗黙的

**影響**: VideoCard、AIBadge、QualityBadge 等の共有コンポーネントが各画面スペックに分散。正規定義の所在が不明確。

> 言及: designer

### 3.7 再レビューのコスト

**影響**: パイプラインでは v1.0 → v2.0 で2ラウンド必要。4ドキュメント全文を2回読む必要があり、コンテキスト消費が大きかった。

> 言及: reviewer

### 3.8 PM の重複メッセージ・権限逸脱

**影響**: tech-leader への重複進捗確認。Gen-4 Turbo 方針決定でPM権限を超えた判断をしようとした。

> 言及: project-manager

## 4. Try（次回試したいこと）

### 4.1 設計着手前のインターフェース契約（Interface Contract）

DB設計者とPerf設計者が着手前に、共通のインターフェース定義（ジョブ型、API パス、enum 値、コスト単価）を先に合意する「契約先行」アプローチ。

> 提案: reviewer, analyzer

### 4.2 外部API調査 → 設計の順序厳密化

researcher の調査完了を設計タスクの前提条件にする。設計着手前に「API 制約確認済み」チェックを設ける。重要制約は速報で即時共有する。

> 提案: tech-leader, designer, researcher

### 4.3 差分レビュー

再レビュー時に全文再読ではなく変更差分のみをレビュー。改訂履歴に変更セクション番号を記載する運用。

> 提案: reviewer

### 4.4 共有コンポーネントライブラリ文書

`docs/figma/shared-components.md` を作成し、VideoCard / AIBadge / QualityBadge 等の正規定義を一元管理。各画面スペックからは参照リンクのみにする。

> 提案: designer

### 4.5 実装フェーズでの Prisma スキーマ先行

全画面の Prisma スキーマを統合した `schema.prisma` を最初に作成し、マイグレーション可能な状態にしてからコンポーネント実装に入る。

> 提案: tech-leader

### 4.6 コスト試算にバッファ係数導入

リトライ・失敗・テスト分を含む「実効コスト = 理論コスト x 1.3〜1.5」の係数を設定。

> 提案: researcher

### 4.7 API 疎通テスト（実装フェーズ初期）

Runway Gen-4 Turbo / Veo 3.1 Fast の最小限 API 呼び出しでレイテンシ・エラーレート・実コストを計測。

> 提案: researcher

### 4.8 レビュー前セルフチェックリスト

設計者がレビュー提出前に、既知の整合性チェックポイント + 機能固有項目を自己確認する。

> 提案: reviewer

### 4.9 パフォーマンスベンチマーク自動化

設計で定めた目標値を CI/CD に組み込み、閾値超過時に自動アラートを発生させる。

> 提案: analyzer
