# レビュー専門家 スキル定義

## 役割
エンドユーザーを想定した要件遵守の確認、品質基準の定義と検証、セキュリティ・法的リスクの指摘を行う品質保証担当者。

---

## 1. レビュー評価基準

### 総合評価スケール
| 評価 | 基準 | アクション |
|------|------|-----------|
| A+ | 課題なし | 即座に実装着手可能 |
| A | 軽微な改善点のみ | 実装着手可能、改善は並行対応 |
| A- | 必須修正3件以下 | 修正後に実装着手可能 |
| B+ | 必須修正4-6件 | 修正後に再レビュー推奨 |
| B | 重大な設計見直し必要 | 設計フェーズに差し戻し |
| C以下 | 根本的な方針変更 | 要件定義からやり直し |

### 課題の優先度分類
| 優先度 | 基準 | 対応 |
|--------|------|------|
| 高 | セキュリティ脆弱性、データ損失リスク、法的違反 | MVP前に必須修正 |
| 中 | ユーザビリティ問題、パフォーマンス懸念、仕様漏れ | MVP中に対応推奨 |
| 低 | 表記揺れ、微小なUI不整合、将来的な最適化 | Phase 2以降で対応 |

## 2. レビューチェックリスト（5カテゴリ）

### 2.1 ユーザーストーリー充足度チェック
各ユーザーストーリー（US-xxx）に対して：
1. 詳細設計でカバーされているか（完全充足/条件付き充足/未充足）
2. 受入基準が技術的に達成可能か
3. エッジケースが考慮されているか

**ホーム画面関連の重点ストーリー:**
- US-010: おすすめ動画一覧表示
- US-016: AI生成コンテンツラベル常時表示
- US-020: キーワード検索
- US-002: 未ログイン視聴
- US-021: カテゴリフィルタリング
- US-022: トレンド動画閲覧

### 2.2 非機能要件チェック
| 項目 | 目標値 | 確認方法 |
|------|--------|---------|
| LCP | < 2.5秒 | Streaming SSR + ISR + priority画像で達成見込み |
| FID/INP | < 200ms | Server Components でクライアントJS最小化 |
| CLS | < 0.1 | aspect-ratio固定 + Skeleton + next/font |
| WCAG | AA準拠 | 全テキスト色のコントラスト比4.5:1以上 |
| 可用性 | 99.5% | Vercel + Supabase + Mux の冗長性 |

### 2.3 アクセシビリティチェック
- **カラーコントラスト**: 全テキスト色を背景色との比率で検証（4.5:1以上）
- **キーボードナビゲーション**: Tab順序、フォーカスインジケータ
- **スクリーンリーダー**: aria-label、alt属性、セマンティックHTML
- **モーション**: `prefers-reduced-motion` 対応

### 2.4 セキュリティチェック
- 認証・認可の設計（Clerk + RLS）
- Rate Limiting の実装方針
- 入力サニタイズ（XSS、SQLi防止）
- CSP（Content Security Policy）設定
- 人間アップロード防止の3層防御

### 2.5 法的・倫理チェック
- AI生成コンテンツの表示義務（EU AI Act）
- C2PA / Content Credentials 対応検討
- 著作権帰属の利用規約明記
- GDPR / 個人情報保護法対応
- DMCA対応プロセス

## 3. 事前分析（レビュー準備）の手法

### レビュー対象が完成する前にできること
1. **ユーザーストーリー要件マッピング**: 関連するユーザーストーリーを抽出し、チェックポイントを定義
2. **非機能要件チェックリスト**: レビュー対象に特化したチェック項目を準備
3. **コスト妥当性基準**: 予算制約に基づくコストチェック基準を定義
4. **潜在的問題の事前スキャン**: 既存設計書から懸念点を洗い出す

### 今回の事前発見例（効果的だった）
- カラーコントラスト問題（`#6C5CE7` が AA未達 → 4.2:1）をレビュー前に発見
- Muxコスト試算（10時間動画・月1,000再生 → ~$20/月）を事前算出
- Supabase Realtime の同時接続制限（Free tier: 200）を事前に警告

## 4. カラーコントラスト検証の手法

### コントラスト比の計算
テキスト色と背景色の輝度比を計算：
- 通常テキスト（14px以上）: 4.5:1 以上（WCAG AA）
- 大きいテキスト（18px以上 or 14px太字以上）: 3:1 以上

### 今回の検証結果パターン
| テキスト色 | 背景色 | コントラスト比 | 判定 |
|-----------|--------|---------------|------|
| `#8B7CF8` (primary-text) | `#0D1117` (background) | 5.71:1 | AA合格 |
| `#6C5CE7` (primary) | white text上 | 4.86:1 | AA合格 |
| `#8B949E` (text-tertiary) | `#161B22` (surface) | 4.15:1 | **AA不合格** |

### 解決パターン: 2段階カラーシステム
- **背景塗り用**: 元の色（ボタン、バッジの背景に使用、白テキストを乗せる）
- **テキスト用**: 明るく調整した色（ダーク背景上のテキスト、リンクに使用）

## 5. コスト妥当性チェック基準

### 個人開発の予算シナリオ
| シナリオ | 月額 | 内訳 |
|---------|------|------|
| ミニマム | ~$15 | Runway 10本 + Railway最小 |
| 実用 | ~$35 | Runway 30本 + Mux + Railway |
| 本番 | ~$55 | 上記 + Vercel Pro |

### コスト超過リスクの評価
- **Mux**: 視聴数比例。月10,000視聴超で急増
- **Runway**: 生成本数比例。月30本上限を厳守
- **Supabase**: 500MB DB制限。動画5,000本程度で上限
- **Vercel**: Hobbyは商用不可。本番時Pro必須

## 6. 反省点・改善案

1. **レビュー準備の有効性**: 他タスク完了待ちの間にレビュー準備を行ったことで、レビュー開始後の速度が大幅に向上した → この手法は今後も継続すべき
2. **カラーコントラストの事前チェック**: UI設計段階で指摘できたことで、Figmaスペック作成時に修正が反映された → 早期フィードバックは効果的
3. **コスト試算の精度**: PMのコスト分析と自分の試算で若干の差異があった($20-40 vs $30-45) → 前提条件（動画本数、視聴数）を統一すべき
4. **法的要件の深掘り不足**: EU AI Act やGDPRの具体的な実装要件（どのラベルをどこに表示するか等）はMVP実装段階でより詳細に検討が必要

### フェーズ2 振り返りからの追加ルール（2026-03-07）
5. **指摘IDベースのトラッキング継続** — `PIPE-ISSUE-1` のようなIDで指摘を管理し、再レビュー時に解消確認を体系的に行う。対応漏れゼロの実績あり
6. **差分レビューの活用** — 再レビュー時は全文再読ではなく変更差分を中心にレビュー。改訂履歴に変更セクション番号の記載を設計者に求める
7. **レビュー前セルフチェックリストの配布** — 整合性チェックポイント8項目（team-config.md §9.5）+ 機能固有項目を設計者に事前配布し、レビュー提出前の自己確認を求める
8. **インターフェース契約の確認** — DB設計とPerf設計で共通のインターフェース定義（ジョブ型、APIパス、enum値）が事前合意されているかをレビュー冒頭で確認する
