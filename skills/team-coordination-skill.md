## 1. チーム構成パターン

### 4エージェント構成（本プロジェクトで検証済み）
| エージェント | subagent_type | 役割 |
|-------------|---------------|------|
| project-manager | general-purpose | タスク管理、整合性チェック、コスト管理 |
| tech-lead | general-purpose | 技術調査、DB設計、パフォーマンス設計 |
| ux-designer | general-purpose | UI/UX設計、Figmaスペック作成 |
| reviewer | general-purpose | 要件レビュー、品質基準、法的チェック |

### エージェント起動時のプロンプト必須要素
1. **役割の明確な定義**
2. **プロジェクト概要**（1-2文）
3. **最初のタスク**（TaskUpdateでアサイン + in_progressにする指示）
4. **チームメンバーの名前と役割**
5. **言語指定**（日本語で作業）
6. **出力先ディレクトリ**

## 2. タスク管理のベストプラクティス

### タスク作成のルール
- 基本設計タスクは1ファイル=1タスクで管理
- 詳細設計タスクは画面単位×観点（UI/UX、DB、パフォーマンス、Figma、レビュー）で分割
- 各タスクに明確な出力先ファイルパスを指定

### 依存関係の設定パターン
```
[UI/UX改善議論] ─────→ [Figmaスペック]
[DB設計]         ─────→ [レビュー]
[パフォーマンス設計] ──→ [レビュー]
[Figmaスペック]  ─────→ [レビュー]
```
- UI/UX改善 → Figma（提案内容を反映するため順次）
- DB・パフォーマンス・Figma → レビュー（全完了後にレビュー開始）
- DB設計とUI/UX改善は並行可能（ただし後で整合性チェック必要）

### タスク完了後のフロー
1. TaskUpdate で completed に変更
2. PMに整合性チェックを依頼
3. 次のタスクをアサインまたはアンブロック通知

## 3. エージェント間コミュニケーション

### 情報共有のルール
- **成果物が上がったら**: PMに即座に整合性チェックを依頼
- **不整合が見つかったら**: 該当担当者に具体的な修正内容を指示（テーブル形式が効果的）
- **技術選定が確定したら**: 全エージェントに確定構成を共有（各エージェントの設計に影響するため）

### 効果的だったコミュニケーションパターン
1. **reviewerの事前準備**: レビュー対象完了待ちの間に準備作業を実施 → レビュー開始後の速度向上
2. **PMのクロスチェック**: 2つの成果物間の不整合を具体的に指摘 → 手戻り最小化
3. **tech-leadへの修正指示**: 修正内容をPrismaコード付きで具体的に指示 → 曖昧さなく反映

### 避けるべきパターン（反省）
1. **同じ質問の繰り返し**: tech-leadに進捗確認を2回送った → 1回で十分、idle通知は正常状態
2. **情報の重複共有**: PMとtech-leadに同じ内容を別々に送った → broadcastか1回の共有で済ませる
3. **ユーザーへの過度な質問**: 最初にAskUserQuestionを3回連続で行い中断された → 1回で必要な情報を聞くか、合理的なデフォルトで進める

- **Figmaスペック**: 画面単位でファイル分割
- **進捗管理**: PROJECT-STATUS.md に集約

## 5. フェーズ移行の判断基準

### 基本設計 → 詳細設計
- 全基本設計タスクが完了
- 技術スタックが確定
- PM整合性チェック完了
- ユーザーの承認

### 詳細設計 → 実装
- 詳細設計のレビュー評価がA-以上
- 高優先度の課題が0件
- 必須修正が反映済み
- コスト見積もりが予算内
- ユーザーの承認

## 6. 全体の反省点・次回への改善

### プロセス改善
1. **技術選定の二転三転を防ぐ**: 初期段階で全選択肢を洗い出し、比較表で一括判断 → 途中変更を最小化
2. **コスト制約の早期共有**: プロジェクト開始時に予算を全メンバーに明示
3. **ドキュメント管理ルールの事前確定**: 出力先、命名規則、ファイル分割方針を最初に決める
4. **UI/UX設計とDB設計の同期**: 並行作業する場合、UI要件の概要を先にDB担当と共有してから着手

### エージェント運用改善
1. **idle通知に過剰反応しない**: エージェントのidle状態は正常。応答がない場合のみ再送
2. **ユーザーへの質問は最小限に**: 合理的なデフォルトで進め、判断ポイントでのみ質問
3. **サブエージェントの調査範囲を明確化**: 重複調査や矛盾する推奨を防ぐ
4. **メッセージの重複送信を避ける**: PMに同じ情報を複数回送らない
